STUDENT  AGILE  TEAM  CHECKLIST   TEAM  SETUP  [  ]  We  have  ONE  Product  Owner  (PO)  who  owns  backlog  ordering  and  clarity.  [  ]  We  have  ONE  Scrum  Master  (SM)  who  runs  the  process  and  removes  impediments.  [  ]  We  are  all  Developers  (build  +  test  +  integrate).  “QA”  is  a  shared  responsibility.  [  ]  We  wrote  a  Team  Working  Agreement  (hours,  response  times,  tools,  conflict  rules).  [  ]  We  agreed  on  a  Definition  of  Done  (DoD)  and  we  enforce  it  for  every  story.   SPRINT  RHYTHM  (repeat  every  sprint)  [  ]  Sprint  Planning  ends  with:  Sprint  Goal  +  selected  stories  +  task  plan.  [  ]  Stand-up  is  timeboxed  and  answers:  progress,  plan,  blockers  (not  a  design  meeting).  [  ]  Backlog  refinement  happens  during  the  sprint  (keep  next  sprint  “ready”).  [  ]  Sprint  Review  shows  a  working  increment  +  demo  evidence  +  feedback  captured.  [  ]  Retrospective  ends  with  1–3  concrete  improvements  we  will  actually  try  next  sprint.   BACKLOG  &  STORIES  [  ]  Every  backlog  item  has:  user  value,  acceptance  criteria,  and  a  size/estimate.  [  ]  User  stories  follow  “As  a…  I  want…  so  that…”  and  meet  INVEST  when  possible.  [  ]  Acceptance  criteria  are  clear,  testable,  and  describe  “what”,  not  “how”.  [  ]  We  track  scope  changes  through  the  backlog,  not  hidden  side  conversations.   ESTIMATION  &  TASK  BREAKDOWN  [  ]  We  estimate  stories  (story  points)  using  Planning  Poker  (or  another  consensus  method).  [  ]  We  break  stories  into  tasks  ≈  0.5–1  day  each  with  owners  and  clear  completion  signals.  [  ]  We  limit  work  in  progress  (finish  before  starting  new  work).   ENGINEERING  RULES  (non ‑ negotiables)  [  ]  Everything  is  in  version  control  (code  +  configs  +  build/test  scripts  +  docs  as  needed).  [  ]  We  integrate  to  trunk/main  frequently  (small  changes;  no  long ‑ lived  branches).  [  ]  CI  runs  an  automated  build  +  fast  tests  often;  broken  builds  are  fixed  immediately.  [  ]  Every  change  gets  review  (PR  review  or  pair  programming)  before  merging.  [  ]  Tests  exist  at  multiple  levels  (many  unit,  some  integration,  few  end ‑ to ‑ end).   DOCUMENTATION  &  COMMUNICATION  [  ]  Each  sprint  has  a  short  release  note  /  changelog  entry.  [  ]  Repo  has  a  README:  setup,  run,  test,  and  “how  to  contribute”.  [  ]  We  communicate  respectfully,  keep  discussions  in  team  channels,  and  document  decisions.   INTEGRITY,  INCLUSION,  PROFESSIONALISM  [  ]  We  credit  sources  and  follow  licence  rules;  we  don’t  copy  other  teams’  work.  [  ]  We  do  not  use  undisclosed  third ‑ party  or  AI-generated  work  in  assessed  deliverables.  [  ]  We  design  inclusively  and  check  accessibility  basics  (e.g.,  keyboard,  contrast,  alt  text).  

[  ]  We  show  up,  communicate  absences  early,  and  rebalance  work  transparently.   EVIDENCE  FOR  ASSESSMENT  [  ]  Our  board/backlog,  commits,  reviews,  tests,  and  sprint  notes  show  how  we  worked.  [  ]  We  do  peer  feedback  calmly  and  fairly;  issues  are  raised  early,  not  at  the  end.
